# Vidyalaya AI data directory

This directory contains the local textbook data used to build the Vidyalaya AI
RAG index. It is the hand-off point between OCR and ingestion:

```text
textbook PDF
    -> OCR (currently run in Google Colab)
    -> page-level OCR JSONL downloaded from Google Drive
    -> page-aware chunks
    -> Gemini embeddings cached as JSONL
    -> Qdrant collection
    -> textbook retrieval used by the application
```

The application does **not** read these files while answering a student. At
runtime it searches the Qdrant collection. The local files are the source data
and rebuild cache for that collection.

## Directory structure

```text
data/
├── data.md                         # This guide
├── download_raw.py                 # Downloads source textbook PDFs
├── download_pro.py                 # Downloads processed OCR JSONL from Drive
├── raw/
│   └── textbooks/
│       ├── scert_odisha/
│       │   ├── class_8/
│       │   │   └── <textbook>.pdf
│       │   └── class_9/
│       └── cbse/
└── processed/
    ├── ocr/
    │   └── <board>/
    │       └── class_<number>/
    │           ├── jsonl/
    │           │   └── <subject>.jsonl
    │           ├── raw/            # Optional raw OCR-engine JSON
    │           │   └── <subject>.json
    │           └── md/             # Optional human-readable OCR output
    │               └── <subject>.md
    └── embeddings/
        └── <board>/
            └── class_<number>/
                └── <subject>_<model>_<dimension>.jsonl
```

For the current SCERT Odisha class 8 dataset, the important locations are:

```text
data/processed/ocr/scert_odisha/class_8/jsonl/
data/processed/embeddings/scert_odisha/class_8/
```

## What belongs in each area

### `raw/`

Store original, unmodified textbook PDFs here. Use this path convention:

```text
data/raw/textbooks/<board>/class_<number>/<book>.pdf
```

This folder may remain empty in a checkout. The present workflow runs OCR in
Colab and keeps the PDFs and OCR working files in Drive. A local PDF is needed
only when OCR is being run or reproduced locally.

### `processed/ocr/`

Store OCR output here, grouped by board and class. The ingestion pipeline reads
only the page-level files inside `jsonl/`. The `raw/` OCR JSON and rendered
Markdown are useful for debugging and review, but are not ingestion inputs.

One `<subject>.jsonl` file represents one book/subject. Every non-empty line is
one JSON object for one page. The required fields are:

| Field | Type | Meaning |
| --- | --- | --- |
| `board` | string | Board key, for example `scert_odisha` |
| `class` | integer | Class number, for example `8` |
| `subject` | string | Subject key matching the filename |
| `book_name` | string | Stable, readable book identifier |
| `book_id` | string | Globally stable ID used to build chunk IDs |
| `language` | string | Language code, for example `or`, `hi`, or `en` |
| `source_pdf` | string | Original PDF filename/source |
| `page_no` | integer | Page number in the source book |
| `text` | string | OCR text for that page |

Example (shortened):

```json
{"board":"scert_odisha","class":8,"subject":"maths","book_name":"ganita_prakash","book_id":"scert_odisha_class_8_maths_ganita_prakash","language":"or","source_pdf":"Maths_Ganita_Prakash.pdf","page_no":1,"text":"...OCR text..."}
```

Empty page text is allowed in the file, but ingestion skips those pages. Board,
class, and subject values must exactly match the ingestion configuration and
the subject filename.

> **Compatibility note:** the current Drive JSONL files contain all required
> fields. At the time of writing, `ocr/convert.py` does not add `book_name`,
> `book_id`, or `language`. Add those fields before using locally converted OCR
> output with `ingestion/main.py`.

### `processed/embeddings/`

These files are generated by ingestion and serve as a resumable embedding
cache. Do not edit them manually. The filename records the subject, embedding
model, and vector dimension:

```text
<subject>_gemini-embedding-2_1536.jsonl
```

Each line contains:

```json
{"chunk_id":"<stable-id>","vector":[1536 numbers],"payload":{"board":"...","class":8,"subject":"...","page_no":1,"chunk_index":1,"text":"..."}}
```

Chunks never cross page boundaries. The current defaults target 2,200
characters, allow up to 3,000 characters, and use a 350-character overlap.
Chunk IDs are deterministic:

```text
<book_id>_p<zero-padded-page>_c<zero-padded-chunk>
```

If ingestion stops partway through, rerunning it reuses records already present
in the embedding cache and embeds only missing chunk IDs.

## Current Colab and Drive workflow

1. Run textbook OCR in Colab.
2. Save the OCR artifacts in Drive. JSON and Markdown can be retained there for
   inspection; the repository ingestion step specifically needs the JSONL.
3. Download each subject JSONL into:

   ```text
   data/processed/ocr/<board>/class_<number>/jsonl/<subject>.jsonl
   ```

   For the currently configured SCERT Odisha class 8 Drive files, run this from
   the repository root:

   ```bash
   python data/download_pro.py
   ```

4. Confirm that each JSONL uses the required schema above. Do not rename a file
   unless its internal `subject` value is changed to match.
5. Set the required credentials in `.env`:

   ```dotenv
   GEMINI_API_KEY=...
   QDRANT_URL=...
   QDRANT_API_KEY=...
   ```

6. Run the complete pipeline from the repository root:

   ```bash
   python -m ingestion.main
   ```

   This loads and validates every configured subject, skips empty pages, creates
   page-aware chunks, writes/reuses the local embedding cache, ensures the
   Qdrant collection exists, and upserts all chunks.

7. If embedding files already exist and only Qdrant needs to be rebuilt or
   updated, avoid calling Gemini again:

   ```bash
   python -m ingestion.upload_embeddings_to_qdrant_cloud
   ```

   This uploads all matching `*_gemini-embedding-2_1536.jsonl` files from the
   configured board/class directory.

## Configuration and adding new data

The current dataset configuration lives in `ingestion/main.py` under
`IngestionConfig`. It presently uses:

- board: `scert_odisha`
- class: `8`
- subjects: `english`, `hindi`, `maths`, `odia`, `sanskrit`, `science`, and
  `social_science`
- embedding model: `gemini-embedding-2`
- vector dimension: `1536`
- Qdrant collection: `vidyalaya_textbook_chunks`

To add another board, class, or subject:

1. Create the matching directory hierarchy under `processed/ocr/`.
2. Put one valid JSONL file per subject in its `jsonl/` directory.
3. Update `IngestionConfig` with the board, class, and subject list.
4. If uploading previously generated embeddings, update `UploadConfig` in
   `ingestion/upload_embeddings_to_qdrant_cloud.py` to the same values.
5. Run ingestion and verify the resulting Qdrant collection.

Keep board and subject keys lowercase and filesystem-safe. Treat `book_id` as
permanent: changing it changes every chunk ID and creates a different set of
Qdrant points.

## Data ownership and version control

The contents of `data/raw/` and `data/processed/` are intentionally ignored by
Git because PDFs, OCR output, and vectors are large generated artifacts. Only
the directory placeholders, helper scripts, and this documentation should be
expected in a fresh clone. Google Drive is currently the external source of
truth for completed OCR artifacts; Qdrant is the deployed retrieval index.

Never place API keys, Drive credentials, or Qdrant credentials in this folder or
commit them. Keep secrets in `.env`, which is also ignored by Git.
